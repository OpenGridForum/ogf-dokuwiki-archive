wiki1934: PhoneMeeting20080207
  • Date: 07 Feb 2008
  • Time: 4 PM Rome (conversion to other timezones)
  • Duration: 1,5h
  • Calling number: +41 22 76 77000 (ask to be connected to "Glue Schema Meeting")
  • Shared screen session information:
    • URL: http://glue-wg.glance.net
    • Key: 0702

Agenda

  1. Storage schema draft
    1. Discuss refined model
    2. Defining attributes from StorageService, StorageEndpoint

Participants:

  • Felix Ehm (CERN)
  • Paul Millar (DESY)
  • Stephen Burke (RAL)
  • Laurence Field (CERN)
  • Sergio Andreozzi (INFN)
  • Marteen Litmath (CERN)
  • JP Navarro (TERAGRID)

Minutes

split document for easy editing and update

change multiplicity of relationship as follows:

  • StorageEndpoint(*)---StorageService
  • StorageEndpoint(*)---(*)StorageShare
  • the main entities need to be updated
  • how do we discover that a storage share can be accessed via file protocol from a certain execution environment?
  • file protocol (meaning POSIX access) implies a relationship between the storage share and the executionEnvironment from which this is available

potential attributes for accessProtocol

  • type
  • version
  • number of streams (for GridFTP)

discussion about several proposals for capturing this:

  • one view is to capture the access protocol in an independent entity
  • another view is to model the access protocol as storage endpoint

the problem raises with SRM

  • for SRM we need to publish the type of access protocols, but not the URL
  • in case of classic SE, we need to publish the URL of the access protocol therefore, both storageEndpoint and AccessProtocol would be instantiated

the idea of the access protocol is to enable discovery of "types of access protocols"; in same case (e.g., SRM) we don't need extra info; in other case, we need extra info (e.g., URL);

Discussion on StorageShare

  • UserTag should be *, we can have many
  • ExpirationMode should be 0..1

Need to add concepts captured by the VOInfo of GLUE 1.3

Discussion on hierarchical shares; use case related to reserved spaces within shares

Decision to map VOInfo concept as specialization of MappingPolicy between UserDomain and StorageShare

Next call: 15/2, 1 PM UK
