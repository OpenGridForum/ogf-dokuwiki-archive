wiki1991: PhoneMeeting20080404
  • Date: 04 April 2008
  • Time: 2 PM CET (conversion to other timezones)
  • Duration: 1,5h
  • Calling number: +41 22 76 77000 (ask to be connected to "GLUE Meeting")

Agenda

  • Suggestion for linking a ComputingService to a StorageService for Glue 1.3 CESEBind replacement
    • no weight attribute
    • mountInfo : how to map the CE path to the SShare path

  • NorduGrid Use Case #7: Provide storage activities (transfer progresses) and related information
    • Is this archievable in the layer of GLUE ?
    • What IS possible to define?

Participants

  • Paul Millar (DESY)
  • Jens Jensen (RAL)
  • Stephen Burke (RAL)
  • Balazs Konya (NorduGrid)
  • Maarten Litmaath (CERN)
  • Felix Ehm (CERN)

Minutes

  • ComputingService to a StorageService Link entity
    • SB : need for a common service service bidirectional relationship class where CE_SE and SE_CE link would inherit from
    • current CESEBind in 1.3 is published by CE - but its information should be split into two parts:
      • one published by the StorageService about link quality of service (SSCSLink)
      • one published by the ComputingService which describes the 'path' which is seen from the ComputingService on the StorageService (CSSSLink)
      • AGREED: add ProtocolLocalID (0..*) which is the id from the AccessProtocol.LocalID
    • JJ: Network connection information between SS and CS could be available in the SSCSLink object
      • AGREED: add NetworkType_t (ComputingResource) in SSCSLink (0..*)
      • AGREED Bandwidth attribute (0..1)
    • SB: example for ACLs mechanism would be than attached to this
      • common opinion : leave this out for next version
    • CSSSLink entity:
      • ID (1)
      • Name (0..1)
      • ID of ComputingService (1)
      • ID of the StorageService (1)
      • Path on the StorageService (1)
      • Path seen from the ComputingService (1)
