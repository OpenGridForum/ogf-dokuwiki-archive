LDAP

DIT

  • Balazs: we need to specify also the DIT
as part of the rendering
  • Laurence:
    • two styles of rendering
      • single entry style
      • multiple entry style
  • Stephen: dn should be called using ID
  • Sergio: dn should be built using classNameAttributeName

Prefix/Namespace

  • Prefix
    • Should be use commont prefix/namespace in the attributes?
    • Balazs: no, OID is the identifier
  • schema A:
    • objectclass: GlueLocation
    • attributeName: Location
  • schema B:
    • objectclass: XXXLocation
    • attributeName: Location

if I configure an LDAP server to load schema A and B location attributes will collide

  • to be investigated by Laurence;
possible solutions:
    • drop prefix from attribute names
and objectclass name
    • add prefix GLUE to both attribute names
and object class names

Representing relationships

  • dn: ComputingServiceID="..", AdminDomainID=""
  • dn: ComputingEndpointID="A", ComputingServiceID="..", AdminDomainID=""
  • dn: ComputingEndpointID="B", ComputingServiceID="..", AdminDomainID=""
  • dn: ComputingShareID="A", ComputingServiceID="..", AdminDomainID=""
  • dn: ComputingShareID="B", ComputingServiceID="..", AdminDomainID=""
  • dn: ComputingResourceID="B", ComputingServiceID="..", AdminDomainID=""
  • dn: ComputingEndpoint_Share="AandB", ComputingServiceID="..", AdminDomainID=""
  • Stephen: at which level we publish the relationships? especially for those belonging to different levels of the DIT or being published by different publishers (i.e. GRIS in gLite)

Resolving relationships

  • about query:
    • Stephen: it would be nice to see the queries to resolve relationships
    • Sergio: possibly require 3 queries
    • Balazs: you can run them at the client side, you get the sub-tree and then check at the client
  • for the rendering, it is still early to work at the attribute level
  • suggestion: extend to design the DIT and the way to model relationships

Back to the GLUE Spec

  • Endpoint Class
    • Stephen: Steve was suggesting that we should
split the Endpoint Class; separate between static and dynamic
    • Sergio: this problem was raised also in the past
      • can be seen in two different ways:
        • should be model and reflected in the conceptual model
        • not a concern of the conceptual model,
concern of implementation
  • Action: verify for each class what are the dynamic attributes
and then evalute if they should be isolated in a different class
    • for endpoint, only state and stateInfo are the dynamic values
to be isolated
  • if we isolate dynamic values in independent classes,
and we stick on mapping a UML class in a different XML element, then this will affect the XML rendering
  • e.g.:
    • <endpoint ID="">
      • <endpointState>
      • </endpointState>
    • </endpoint>
  • Capability:
    • Stephen: do we keep capability both here and in service?
    • Sergio: will provide initial enumeration for both then we discuss again

  • ServingState: open, close
  • AcceptingState: open, close

vs.

  • ServingState: production, draining, closed, queueing
  • Stephen:
  • Suggest downtime info go in a different object
  • evalute if 1-to-many relationship should apply
    • an endpoint having multiple scheduled downtime
  • TrustedCA should go together with access control information

Action

  • Balazs: to check if UR WG adopt dateType XML Schema type or POSIX timestamp
  • Laurence: investigate the pro/con of having prefix in ldap attributes/classes names
  • Sergio: draft list of service/endpoint capabilities
