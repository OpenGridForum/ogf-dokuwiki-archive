  during the requirements analyses for the OCCI working group. They are used

  to set up the requirements and later on to verify the OCCI

  specification.</p><div class="section" xml:lang="en"><div class="titlepage"><div><div><h2 class="title"><a id="id440973"/>SLA-aware cloud infrastructure using SLA@SOI</h2></div></div></div><p>There is a need for a standard interface for dynamic infrastructure
7 
    provisioning. While doing so it must be guaranteed and verified that the
8 
    infrastructure provisioning uses 'machine-readable' SLAs. <span class="citerefentry"><span class="refentrytitle">(SLA_SOI)</span></span></p><h3><a id="id440991"/>Functional Requirements</h3><div class="itemizedlist"><ul><li><p>VM Description: request format important - In this area is where
9 
        there is least coherency amongst providers.</p></li><li><p>VM Description: a means to add non-functional constraints on
10 
        functional attributes.</p></li><li><p>VM Management: all parameters in the request should be
11 
        "monitor-able" and verifiable. Full control of resources (VMs)
12 
        allocated required; at a minimum: start, stop, suspend, resume.</p></li><li><p>VM Monitoring: Monitoring non-functional constraints declared in
13 
        provisioning request</p></li><li><p>Network Management: resources assignable by network tag -

        defaults of public and private further sub-categorisation could be

        allowed e.g. tag of web could be assigned to the public network
16 
        group.</p></li><li><p>Storage Management: simple mount points, reuse storage SaaS
17 
        offerings</p></li></ul></div><h3><a id="id441050"/>Non-functional Requirements</h3><div class="itemizedlist"><ul><li><p>Security: Transport and user level (ACLs? oAuth?)
18 
        security</p></li><li><p>Quality of Service: Can be many - Part of service offering from
19 
        the infrastructure provider e.g. Security, QoS, geo-location,
20 
        isolation levels - NFPs are the basic building blocks of
21 
        differentiating IaaS providers.</p></li><li><p>Scheduling Information: When a particular resource is to be run.
22 
        Also in which order should a collection of resources be ran in the
23 
        case that one resource is dependent on another.</p></li></ul></div></div><div class="section" xml:lang="en"><div class="titlepage"><div><div><h2 class="title"><a id="id441086"/>Service Manager to control the Life cycle of Services</h2></div></div></div><p>This use case is based in the 'Service Manager' (SM) layer of the
24 
    RESERVOIR project architecture. 'Service Providers' (SP) willing to deploy
25 
    their service on the Cloud use this layer to control the service life
26 
    cycle. The SM operates over the Cloud infrastructure automatically as the
27 
    service demands. In a way, the SM maps the service configuration and needs
28 
    to calls to the Cloud infrastructure, so many of the requirements imposed
29 
    by the SM are due to the flexibility that the SM aims to provide to SPs.
30 
    (<span class="citerefentry"><span class="refentrytitle">RV</span></span>)</p><h3><a id="id441102"/>Functional Requirements</h3><div class="itemizedlist"><ul><li><p>Network Management: There should be methods for the Allocation
31 
        of private networks, where VMs can be attached to. A special network
32 
        (e.g. 'Public Network') should be available. When some network
33 
        interface is attached to it, the infrastructure must assign a public
34 
        IP address.</p></li><li><p>Image Management: There should be methods to register, upload,
35 
        update and download disk images.</p></li><li><p>VM Description: It should be possible to describe all the VM
36 
        hardware components and their attributes, along with any restriction
37 
        regarding the VM location:</p><div class="itemizedlist"><ul><li><p>Memory: Size</p></li><li><p>CPU: Architecture, amount of CPU's and speed.</p></li><li><p>Disk: Size, Interface (SCSI, IDE, SATA...), RAID (yes/no,
38 
            and RAID level), Disk image to mount, Automatic backup (yes/no,
39 
            backups frequency...).</p></li><li><p>Network: Interfaces, for each interface its bandwidth, and
40 
            Network they are attached to.</p></li><li><p>Geographical restrictions: Location(s) where the VM
41 
            can/cannot be deployed (for example for legal purposes).</p></li><li><p>Migration allowed (yes/no): If migration is supported by the
42 
            infrastructure, this flag sets if it is allowed for the VM.</p></li></ul></div></li><li><p>VM Management: There should be methods to allow the SM to change
43 
        the VM state (for example, from ACTIVE to SUSPENDED), if such
44 
        transition is allowed by the infrastructure (i.e. is defined in the
45 
        OCCI's State Machine). The description of a VM can be changed when the
46 
        machine is running (ACTIVE, SUSPENDED...). But it will not be taken
47 
        into account until the machine is stopped and started again, unless it
48 
        is a change regarding geographical or migration restrictions. Each
49 
        disk backup will have an id, as the images defined by the SM. Methods
50 
        to download any backup should be provided. As each backup is, after
51 
        all, a disk image, it should be possible to mount it on any VM. For
52 
        example, it should be possible to stop a VM, change its configuration
53 
        so its disk mounts this backup image, and restart the VM.</p></li><li><p>Monitoring: The status (We use the term 'status' when talking
54 
        about monitoring, and try not to use the term 'state' to avoid
55 
        confusion with the states of the OCCI State Machine.) representation
56 
        of any element is given as a list of keys and their values. For
57 
        example, the status of a memory component could be given by the amount
58 
        of memory used and the cache memory. Then, the keys could be: 'used'
59 
        and 'cache' with the values '142MB' and '430MB'. Both the request and
60 
        the reply use the corresponding element identifier. Two types of
61 
        monitoring should be supported:</p><div class="itemizedlist"><ul><li><p>Pull based: The SM can request the status of any element it
62 
            has registered: VMs, networks... Also, the SM can request the
63 
            status of components, for example, the status of certain disk of a
64 
            certain VM.</p></li><li><p>Publish/subscribe based: The SM can subscribe to be notified
65 
            about events on the VMs and/or Networks. Some of the events to be
66 
            notified are:</p><div class="itemizedlist"><ul><li><p>Errors on some component of a VM.</p></li><li><p>Changes on the state of a VM (e.g. from ACTIVE to
67 
                SUSPENDED).</p></li><li><p>Periodic notifications about some element state. The
68 
                frequency of this notifications can be configured in the
69 
                subscription message.</p></li></ul></div></li></ul></div></li><li><p>Error messages: If a VM could not be created, or a image could
70 
        not be uploaded, etc... the platform should return an error message
71 
        carrying a detailed description of the reason.</p></li><li><p>Identifications: Networks, VMs and images should have unique
72 
        IDs, (UUIDs, URIs, or the like). It is to be determined whether
73 
        components of VMs (disks, memory...) should have an unique ID too. IDs
74 
        are assigned by the Cloud infrastructure when the corresponding
75 
        element is created.</p></li></ul></div><h3><a id="id441290"/>Non-functional Requirements</h3><div class="itemizedlist"><ul><li><p>Both for hardware configuration and monitoring values there
76 
        should be a clear, standard way to set which magnitude the value
77 
        represents. For example, when setting the memory size to '2', it must
78 
        be clear that we refer to GBs and not to MBs. An option would be
79 
        setting the value to '2GB', another would be allowing to set both the
80 
        value and the magnitude: value '2' and magnitude 'GB'.</p></li><li><p>Protocols: The transport, message format, and state
81 
        representation should use open and standard protocols, each one which
82 
        strong software support (i.e. libraries and frameworks available for
83 
        several programming languages).</p></li></ul></div></div><div class="section" xml:lang="en"><div class="titlepage"><div><div><h2 class="title"><a id="id441325"/>Interoperability across Cloud Infrastructures using
84 
    OpenNebula</h2></div></div></div><p>OpenNebula is a Virtual Infrastructure Engine, being enhanced in the
85 
    RESERVOIR project, which allows the management of Virtual Machines on a
86 
    pool of physical resources It offers three main functionalities: backend
87 
    of a public cloud, manage a virtual infrastructure in the data-center or
88 
    cluster (private cloud), achieve cloud interoperation (hybrid cloud), the
89 
    latter being relevant in this use case.</p><p>The aim of this use case is to state the requirements that an API
90 
    for cloud providers should take into account in order to expose an
91 
    interface that will enable the management of groups of Virtual Machines
92 
    across them. These requirements are gathered from the experience using
93 
    OpenNebula to manage Virtual Machines from different cloud providers.
94 
    Currently, there are two set of plugins for OpenNebula to access Amazon
95 
    EC2 and ElasticHosts cloud providers that leverage the use of both cloud
96 
    providers in a transparent fashion for the end user. (<span class="citerefentry"><span class="refentrytitle">ONE</span></span>)</p><h3><a id="id441354"/>Functional Requirements</h3><div class="itemizedlist"><ul><li><p>VM Description: Virtual Machines should be described
97 
        consistently across cloud providers using a slim set of indispensable
98 
        attributes, such as:</p><div class="itemizedlist"><ul><li><p>Memory: Amount of RAM needed by the Virtual Machine</p></li><li><p>CPU: Number of CPUs needed by the Virtual Machine (this
99 
            needs to be normalized)</p></li><li><p>Disk: Disks that will conform the basic filesystem and
100 
            possibly others for the Virtual Machine</p></li><li><p>Network: How many network interface this Virtual Machine
101 
            should have, and where should be attached</p></li></ul></div></li><li><p>VM Management: API should offer functionality to enforce
102 
        operations upon Virtual Machines, such as:</p><div class="itemizedlist"><ul><li><p>DEPLOY: Launches the Virtual Machine</p></li><li><p>SHUTDOWN: Shutdown the Virtual Machine</p></li><li><p>CANCEL: Cancels the Virtual Machine in case of failure, or
103 
            destroys it if it is running</p></li><li><p>CHECKPOINT: Creates a snapshot of the Virtual Machine</p></li><li><p>SAVE: Creates a snapshot of the Virtual Machine AND suspends
104 
            it</p></li><li><p>RESTORE: Resumes a Virtual Machine from a previous
105 
            snapshot</p></li><li><p>POLL: Retrieves information about Virtual Machine state and
106 
            consumption attributes (percentage of Memory, CPU used, bytes
107 
            transferred, and so on)</p></li></ul></div></li><li><p>Additionally, Virtual Machines should be in one of the following
108 
        states:</p><div class="itemizedlist"><ul><li><p>PENDING: VM is waiting for a physical resource slot.</p></li><li><p>BOOTING: VM is being booted</p></li><li><p>RUNNING: VM is active, it should be able to start offering a
109 
            service</p></li><li><p>SUSPENDED: VM is suspended, waiting for a resume.</p></li><li><p>SHUTDOWN: VM is being shutdown.</p></li><li><p>CANCEL: VM has been canceled by the user or by a
110 
            scheduler.</p></li><li><p>FAILED: VM crashed or hasn't started properly.</p></li></ul></div></li><li><p>Network Management: API should expose functionality to</p><div class="itemizedlist"><ul><li><p>Create Private Virtual Networks</p></li><li><p>Attach Public IP to Virtual Machine</p></li></ul></div></li><li><p>Image Management: The ability to upload disk images is
111 
        fundamental to virtual machine management to avoid the need to
112 
        reinstall software for each cloud provider. The upload process should
113 
        return an identifier to be used in the Virtual Machine
114 
        Description.</p></li></ul></div><h3><a id="id441594"/>Non-functional Requirements</h3><div class="itemizedlist"><ul><li><p>Security: Security should be handled using X509 certificates for
115 
        authentication. Also, authorization can be based on said certificates
116 
        and ACL lists.</p></li><li><p>Quality of Service: When used in conjunction with Haizea,
117 
        OpenNebula provides advanced reservation functionality. Cloud
118 
        providers API should provide similar capabilities to ensure proper
119 
        QoS.</p></li></ul></div></div><div class="section" xml:lang="en"><div class="titlepage"><div><div><h2 class="title"><a id="id441624"/>AJAX web front-end directly calling API</h2></div></div></div><p>This use case describes the ability to create web front-ends for
120 
    Clouds. A cloud provider implements their customer web front-end as an
121 
    entirely client-side AJAX application calling the OCCI API
122 
    directly.</p><h3><a id="id441638"/>Functional Requirements</h3><div class="itemizedlist"><ul><li><p>Completeness: API must be contain complete set of calls to
123 
        completely specify and control cloud (but this is likely only ~15-20
124 
        verbs on ~3-4 nouns!)</p></li><li><p>Responsiveness: Calls must return swiftly. In particular, we
125 
        should provide a simple and quick call to poll the _list_ of servers,
126 
        drives, etc. that exist without listing all of their properties, since
127 
        this is computationally much cheaper for the cloud to return, and will
128 
        need to be regularly polled to catch any servers, etc. that are
129 
        created outside of the interface.</p></li></ul></div><h3><a id="id441670"/>Non-functional Requirements</h3><div class="itemizedlist"><ul><li><p>Syntax: A simple JSON syntax for the API will make the AJAX
130 
        interface much simpler to implement</p></li></ul></div></div><div class="section" xml:lang="en"><div class="titlepage"><div><div><h2 class="title"><a id="id441688"/>Single technical integration to support multiple service
131 
    providers</h2></div></div></div><p>Today, each cloud provider (ElasticHosts, GoGrid, Amazon, etc.)
132 
    integrates independently with every other player in the cloud ecosystem
133 
    (CohesiveFT, RightScale, etc), producing O(n^2) separate technical
134 
    integrations. In the future, if all cloud providers and cloud ecosystem
135 
    partners use a single standard API, then we have O(n) technical
136 
    integrations, and all potential partnerships can immediately
137 
    interoperate.</p><h3><a id="id441706"/>Non-functional Requirements</h3><div class="itemizedlist"><ul><li><p>Uptake: Standardized IaaS API needs strong uptake in by both

        cloud providers and cloud ecosystem.</p></li></ul></div></div><div class="section" xml:lang="en"><div class="titlepage"><div><div><h2 class="title"><a id="id441724"/>Wrapping EC2 in OCCI</h2></div></div></div><p>Amazon EC2 is the de-facto standard for IaaS today. While Amazon is

    OCCI API would be capable of interfacing to EC2 so that gateways can be
141 
    written and so that Amazon could join at a later date.</p><h3><a id="id441739"/>Functional Requirements</h3><div class="itemizedlist"><ul><li><p>Semantics: Must include the ability to fully describe core EC2
142 
        objects and operations</p></li></ul></div><h3><a id="id441755"/>Non-functional Requirements</h3><div class="itemizedlist"><ul><li><p>A gateway to support the integration of OCCI and EC2</p></li></ul></div></div><div class="section" xml:lang="en"><div class="titlepage"><div><div><h2 class="title"><a id="id441773"/>Automated Business Continuity and Disaster Recovery</h2></div></div></div><p>Maintain a up-to-date remote shadows of physical and/or virtual
143 
    machines, such that in the event of a disaster it is possible to start and
144 
    switch to the remote machines.</p><h3><a id="id441786"/>Functional Requirements</h3><div class="itemizedlist"><ul><li><p>VM Description: Metadata mapping to legacy systems</p></li><li><p>VM Management: Automated management in the event of a disaster
145 
        (e.g. startup, IP changes).</p></li><li><p>Network Management: Runtime alteration of IPs</p></li><li><p>Image Management: Advanced, rsync style updates to synchronise
146 
        machines with physical equivalents (e.g. rsync block devices to remote
147 
        raw disk files).</p></li></ul></div><h3><a id="id441829"/>Non-functional Requirements</h3><div class="itemizedlist"><ul><li><p>Quality of Service: Reservation of capacity sufficient for fail

149

150

151 
    Unix/Linux setup, so would like to write a simple cron job which carries
152 
    this out.</p><h3><a id="id441862"/>Non-functional Requirements</h3><div class="itemizedlist"><ul><li><p>Syntax: This should be as simple as possible to place minimal

        of shell script, posting a simple &lt;5 lines of command data using
156 
        curl, wget, etc.</p></li></ul></div></div><div class="section" xml:lang="en"><div class="titlepage"><div><div><h2 class="title"><a id="id441884"/>Typical web hosting cluster</h2></div></div></div><p>An end-user runs a typical web hosting cluster on a cloud, with: n

158 
    load balancer (either a specialized virtual machine or provided by the
159 
    cloud like GoGrid).</p><h3><a id="id441898"/>Functional Requirements</h3><div class="itemizedlist"><ul><li><p>Completeness: The API should be able to fully express this

    web front-end referred to in this use case: AJAX web front-end directly
167 
    calling API</p><h3><a id="id441936"/>Functional Requirements</h3><div class="itemizedlist"><ul><li><p>Completeness: Every resource provided by the cloud is

173

175 
        need to be regularly polled to catch any servers, etc. that are

        created outside of the interface. (text copied from AJAX web front-end

181 
        example, the client must be able to differentiate between a compute

192 
        terminology defined by the user for his specific context. For example,
193

194 
        "database server", and the resource "/storage/disk12345678" with the
195 
        label "superSecretCorporateData".</p></li><li><p>Searchability: The ability to request lists of resources must
196 
        allow an optional filter that can specify a category or tag upon which
197 
        to filter the results. This allows one to further limit their view to,
198 
        for example, resources tagged "productionEnvironment", or resources of
199 
        the category "storage".</p></li></ul></div><h3><a id="id441999"/>Non-functional Requirements</h3><div class="itemizedlist"><ul><li><p>Usability: This should be a user interface with context-menus

        and context-aware links that allow the user to easily see what actions

        can be performed for each resource.</p></li></ul></div></div><div class="section" xml:lang="en"><div class="titlepage"><div><div><h2 class="title"><a id="id442018"/>Compute Cloud</h2></div></div></div><p>A cloud provider implements a RESTful API for provisioning,

    executing, and monitoring of tasks.</p><h3><a id="id442030"/>Functional Requirements</h3><div class="itemizedlist"><ul><li><p>Secure: API must be secured to ensure that only authorized
203 
        identities are permitted to use the API.</p></li><li><p>Resource: An endpoint must be created for external monitoring,
204 
        status, and auditing of the task. This endpoint would be responsive to
205 
        RESTful calls supporting AJAX and other clients.</p></li><li><p>Scripted: The target system needs to understand and process
206 
        directives which would be provided with the task. These directives
207 
        would include the ability to pull binaries or data onto the system,
208 
        run executables, and status the system resources.</p></li></ul></div><h3><a id="id442069"/>Non-functional Requirements</h3><div class="itemizedlist"><ul><li><p>Single Compute Method: The resultant service should be the same
209 
        service that can be used for many other purposes. It could be used for
210 
        monitoring of system health, system life-cycle management, system
211 
        patching, and configuration changes. If this was the only service on
212 
        the system initially, it could then be used to build up the other
213 
        services in a plug-in manner.</p></li></ul></div></div><div class="section" xml:lang="en"><div class="titlepage"><div><div><h2 class="title"><a id="id442092"/>Multiple Allocation</h2></div></div></div><p>Allocate a whole cluster with one call.</p><h3><a id="id442103"/>Functional Requirements</h3><div class="itemizedlist"><ul><li><p>Definition of groups: There should be a way to define groups of
214 
        computers. In the example of a cluster, there would be two groups: The
215 
        Headnode and a couple of Workernodes.</p></li><li><p>Information: For configuration of the members of the defined
216 
        groups, there should be way (maybe a URL) to find out about all groups
217 
        and their basic configurations. In the example, the Headnode would
218 
        want to know IPs or Hostnames of all Workernodes. The workernodes will
219 
        need to know this, as well _and_ they need to know, that the headnode
220 
        is in a different group.</p></li></ul></div></div><div class="section" xml:lang="en"><div class="titlepage"><div><div><h2 class="title"><a id="id442136"/>Cloud Consumer Discovery of Cloud Provider's VM Input and Output

222 
    formats accepted and delivered by the cloud provider.</p><h3><a id="id442149"/>Functional Requirements</h3><div class="itemizedlist"><ul><li><p>The provider supplies an API which is availed over unsecured
223 
        network connections.</p></li><li><p>The provider supplies an API which is availed over secured
224 
        network connections.</p></li><li><p>The provider supplied API is availed for all consumer
225 
        authentication and authorization levels.</p></li><li><p>The provider supplied API identifies the supported VM input
226 
        formats API uniquely and commonly across all providers.</p></li><li><p>The provider supplied API identifies the supported VM output
227

309

311

312

313

314

315

316

317

318

319

320

321

322

323

328

329

330

331
