\subsection{Network}


    		
  

  

  

    
      11
    
    
      
This document explicitly does not try to provide a definition of the term `Network'.


    		
  

  

  

    
      12
    
    
      
The working group has discussed the possible meanings of this term and in the end we were forced to conclude that is not possible to provide a workable definition for the term \emph{Network}.


    		
  

  

  

    
      13
    
    
      
The term \emph{Network} has become so widely used for so many diverse meanings 


    		
  

  

  

    
      14
    
    
      
that it is impossible to create a definition that everyone can agree on, while still expressing something useful.

\subsection{Network Object}\label{network_object}


    		
  

  

  

    
      17
    
    
      

    		
  

  

  

    
      18
    
    
      
The basic abstract element of the schema is the \emph{Network Object}.  Other


    		
  

  

  

    
      19
    
    
      
basic elements inherit from it.  The \emph{Network Object} can have a


    		
  

  

  

    
      20
    
    
      
\emph{Location}, can be related to other instances via a \emph{Relation} and can be described by a \emph{Lifetime}.  Every Network Object MUST have an \emph{id} attribute, which MUST be a unique URI.  These characteristics are inherited by the subclasses of the \emph{Network Object} class.  


    		
  

  

  

    
      21
    
    
      

    		
  

  

  

    
      22
    
    
      
The base \emph{Network Object} has three related objects that describe the \emph{Network Object} and its relationships:


    		
  

  

  

    
      23
    
    
      

    		
  

  

  

    
      24
    
    
      
\begin{itemize}


    		
  

  

  

    
      25
    
    
      
  \item Location


    		
  

  

  

    
      26
    
    
      
  \item Lifetime


    		
  

  

  

    
      27
    
    
      
  \item Relation


    		
  

  

  

    
      28
    
    
      
\end{itemize}


    		
  

  

  

    
      29
    
    
      

    		
  

  

  

    
      30
    
    
      

    		
  

  

  

    
      31
    
    
      
The location of an object in the physical world can be described using the


    		
  

  

  

    
      32
    
    
      
\emph{Location} object. The actual location is then described using properties


    		
  

  

  

    
      33
    
    
      
of the \emph{Location} object.


    		
  

  

  

    
      34
    
    
      
%\emph{locatedAt} relationship


    		
  

  

  

    
      35
    
    
      

    		
  

  

  

    
      36
    
    
      
All \emph{Network Objects} can potentially have a \emph{Lifetime}, that consists of vector of \emph{time} elements, which contain a start time and an end time.


    		
  

  

  

    
      37
    
    
      

    		
  

  

  

    
      38

      39
    
    
      

    		
  

  

  

    
      40
    
    
      
The base \emph{Network Object} is subclassed into the top-level topology


    		
  

  

  

    
      41

      42
    
    
      
top-level network objects in this schema are:


    		
  

  

  

    
      43
    
    
      

    		
  

  

  

    
      44
    
    
      
\begin{itemize}


    		
  

  

  

    
      45
    
    
      
    \item Node


    		
  

  

  

    
      46
    
    
      
    \item Port


    		
  

  

  

    
      47
    
    
      
    \item Link


    		
  

  

  

    
      48
    
    
      
    \item Service


    		
  

  

  

    
      49
    
    
      
    \item Group


    		
  

  

  

    
      50
    
    
      
\end{itemize}


    		
  

  

  

    
      51

These objects are described in more detail below.

\subsection{Node}


    		
  

  

  

    
      58
    
    
      

    		
  

  

  

    
      59
    
    
      
A \emph{Node} is generally a device connected to, or part of, the network.  A


    		
  

  

  

    
      60
    
    
      
Node does not necessarily correspond to a physical machine.  It may be a 


    		
  

  

  

    
      61
    
    
      
virtual device or a group of devices.


    		
  

  

  

    
      62
    
    
      

    		
  

  

  

    
      63
    
    
      
In this case, the \emph{implemented by} relation can be used to describe the 


    		
  

  

  

    
      64
    
    
      
virtualization.


    		
  

  

  

    
      65
    
    
      

    		
  

  

  

    
      66
    
    
      
% <!-- We first used a Virtual and Physical subclasses. For simplicity reasons


    		
  

  

  

    
      67
    
    
      
% we decided not to use them. Another advantage is that the schema can now


    		
  

  

  

    
      68
    
    
      
% describe an arbitrary long chain of implementedBy's. -->


    		
  

  

  

    
      69
    
    
      

    		
  

  

  

    
      70
    
    
      
A Node is connected to the network by its \emph{Ports}.  A Node can provide 


    		
  

  

  

    
      71
    
    
      
\emph{Services}.  


    		
  

  

  

    
      72
    
    
      

    		
  

  

  

    
      73
    
    
      

    		
  

  

  

    
      74
    
    
      
The Relations of Node:


    		
  

  

  

    
      75
    
    
      

    		
  

  

  

    
      76
    
    
      
\begin{itemize}


    		
  

  

  

    
      77
    
    
      

    		
  

  

  

    
      78
    
    
      
\item A \emph{Node} MAY share a \emph{has port} relation with one or more \emph{Ports}. 


    		
  

  

  

    
      79
    
    
      
\item A \emph{Node} MAY share a \emph{located at} relation with one


    		
  

  

  

    
      80
    
    
      
  \emph{Location} 


    		
  

  

  

    
      81
    
    
      
\item A \emph{Node} MAY share an \emph{implemented by} relation with one or more \emph{Nodes}.  


    		
  

  

  

    
      82

\end{itemize}

\subsection{Port}


    		
  

  

  

    
      86
    
    
      

    		
  

  

  

    
      87
    
    
      

    		
  

  

  

    
      88
    
    
      
A \emph{Port}, or interface, connects a \emph{Node} or \emph{Group} to the rest of the network. A \emph{Port} object is unidirectional.


    		
  

  

  

    
      89
    
    
      

    		
  

  

  

    
      90
    
    
      
A \emph{Port} is related to zero or one \emph{Node} or \emph{Group}, and also


    		
  

  

  

    
      91
    
    
      
has a relation with zero, or one (uni-directional) \emph{Links}.


    		
  

  

  

    
      92
    
    
      

    		
  

  

  

    
      93
    
    
      
To adapt traffic to and from different layers, an \emph{adaptation} relation is used. An adaptation consists of two unidirectional components, an \emph{adaptation sink} to go from a server layer port to a client layer port, and an \emph{adaptation source} for the other direction.


    		
  

  

  

    
      94
    
    
      

    		
  

  

  

    
      95
    
    
      
A Port can have up to two adaptation sink relations, one for a server layer, and one for a client layer. The same applies for adaptation source relations. 


    		
  

  

  

    
      96
    
    
      

    		
  

  

  

    
      97
    
    
      
Relations of Port:


    		
  

  

  

    
      98
    
    
      
\begin{itemize}


    		
  

  

  

    
      99
    
    
      
    \item A \emph{Port} MAY have one \emph{adaptation sink} relations.


    		
  

  

  

    
      100
    
    
      
    \item A \emph{Port} MAY have one \emph{adaptation source} relations.


    		
  

  

  

    
      101
    
    
      
    \item A \emph{Port} MAY have one \emph{source} or \emph{sink} relation with a \emph{Unidirectional Link}.


    		
  

  

  

    
      102
    
    
      
\end{itemize}

\subsection{Link}


    		
  

  

  

    
      106
    
    
      

    		
  

  

  

    
      107
    
    
      
A \emph{Link} object describes that there is a unidirectional connection from one \emph{Port} to another. These ports are identified using the \emph{source} and \emph{sink} relationships.


    		
  

  

  

    
      108
    
    
      

    		
  

  

  

    
      109
    
    
      
A \emph{Link} MUST have an attribute \emph{type} which is either \emph{Link} or \emph{Crossconnect}. When the type is Crossconnect, the source and sink MUST be part of the same node.


    		
  

  

  

    
      110
    
    
      

    		
  

  

  

    
      111
    
    
      
A \emph{Link} should have a \emph{capacity}  attribute which describes the capacity of the link in bytes per second. This value should correspond to the actual amount of data that can be transported over the link, excluding overhead.


    		
  

  

  

    
      112
    
    
      

    		
  

  

  

    
      113
    
    
      
Relations of Link:


    		
  

  

  

    
      114
    
    
      

    		
  

  

  

    
      115
    
    
      
\begin{itemize}


    		
  

  

  

    
      116
    
    
      
    \item A \emph{Link} MAY have a \emph{source} relation with one \emph{Port}.


    		
  

  

  

    
      117
    
    
      
    \item A \emph{Link} MAY have a \emph{sink} relation with one \emph{Port}.


    		
  

  

  

    
      118
    
    
      
    \item A \emph{Link} SHOULD have a \emph{capacity} attribute which describes the capacity of the link in \emph{bytes per second}.


    		
  

  

  

    
      119
    
    
      
\end{itemize}

\subsection{Service}


    		
  

  

  

    
      122
    
    
      

    		
  

  

  

    
      123
    
    
      
A \emph{Service} object describes a certain capability being offered by a


    		
  

  

  

    
      124
    
    
      
Network Object.  The key idea that this is a generic container representing 


    		
  

  

  

    
      125
    
    
      
any service that a network-centric user or agent might want to discover and 


    		
  

  

  

    
      126
    
    
      
use. Below we describe some example Services, but others are also possible.


    		
  

  

  

    
      127
    
    
      

    		
  

  

  

    
      128
    
    
      

    		
  

  

  

    
      129
    
    
      
\begin{description}


    		
  

  

  

    
      130
    
    
      
    \item[SwitchingMatrix] describes the ability of a network object to create cross connects between its different ports.


    		
  

  

  

    
      131
    
    
      
    


    		
  

  

  

    
      132
    
    
      
     Configured cross-connects should be described using the \emph{switched to} relation.


    		
  

  

  

    
      133
    
    
      
    \item[Adaptation] describes that ports on different layers within a node can possibly be connected together to form a connection or cross connect on a different layer. The Adaptation service must define both its client and server layer.


    		
  

  

  

    
      134
    
    
      
    


    		
  

  

  

    
      135
    
    
      
    Once this is implemented it is described using the \emph{adaptation source} and \emph{adaptation sink} relations between ports.


    		
  

  

  

    
      136
    
    
      

    		
  

  

  

    
      137
    
    
      
    \item[Measurement Point] services are an essential component of network measurement services like perfSONAR.


    		
  

  

  

    
      138
    
    
      

    		
  

  

  

    
      139
    
    
      
\end{description}

      169
    
    
      

    		
  

  

  

    
      170
    
    
      
A \emph{Topology} is a set of Network Objects and the links connecting them.


    		
  

  

  

    
      171

\subsubsection{Path}


    		
  

  

  

    
      175

      177

\begin{description}


    		
  

  

  

    
      210

\subsection{Relation}\label{sub:relation}


    		
  

  

  

    
      220
    
    
      

    		
  

  

  

    
      221
    
    
      
\emph{Relations} describe how different network objects can be combined to form a network topology description. The relations have been described above, but for ease of reference we also give a full list and definition here (in alphabetical order):


    		
  

  

  

    
      222
    
    
      

    		
  

  

  

    
      223
    
    
      
\begin{description}


    		
  

  

  

    
      224
    
    
      
    \item[adaptation] is the bi-directional equivalent of the source and sink adaptation combination.    


    		
  

  

  

    
      225
    
    
      
    \item[adaptation sink] goes from a server layer port to a client layer port, describing the way that data is passed between these two layers.


    		
  

  

  

    
      226
    
    
      
    \item[adaptation source]  is the reverse of the sink adaptation, i.e. it goes from a client layer port to a server layer port.


    		
  

  

  

    
      227
    
    
      
    \item[at layer] is a relation between a port and a layer to describe the layer at which the port operates.    


    		
  

  

  

    
      228
    
    
      
    \item[has port] describes the relation between a node or group and a port.


    		
  

  

  

    
      229
    
    
      
    \item[has service] describes the relation between a node or group and a service that it provides.


    		
  

  

  

    
      230
    
    
      
    \item[implemented by] is a relation from a node to a node, describing that the source node is a virtualized node on the sink node of the relation.


    		
  

  

  

    
      231
    
    
      
    \item[located at] is a relation between a network object and a location object.


    		
  

  

  

    
      232
    
    
      
    \item[sink] is the connection between the end of the link and the sink port.


    		
  

  

  

    
      233
    
    
      
    \item[source] describes the connection of a port to the source port of the link.


    		
  

  

  

    
      234
    
    
      
    \item[switched to] defines a relation between two ports, meaning that traffic from the source port is automatically forwarded to the destination port.


    		
  

  

  

    
      235
    
    
      
\end{description}
