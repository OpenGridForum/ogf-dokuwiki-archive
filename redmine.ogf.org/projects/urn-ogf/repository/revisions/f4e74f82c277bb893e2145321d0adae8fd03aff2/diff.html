\section*{Abstract}


  		





  
  139
  
    

  		





  
  140
  
    
This document specifies URNs in the \texttt{urn:ogf:network} namespace. URNs in this namespace can be used to define logical network resources, such a devices, (logical) ports, and (logical) links.

\section{introduction}%


  		





  
  148
  
    
\label{sec:introduction}


  		





  
  149
  
    

  		





  
  150
  
    
Uniform Resource Names (URNs) are persistent, globally unique identifiers~\cite{rfc2141}.


  		





  
  151
  
    

  		





  
  152
  
    
Topology exchange between network operators requires globally unique identifiers for network resources. The \texttt{urn:ogf:network} namespace provides globally unique identifiers for naming network resources without central registration.  This document defines and registers the \texttt{urn:ogf:network} namespace in accordance with \cite{urn-ogf-procedure}.

\section{Syntax}

\subsection{Syntactic Structure}


  		





  
  191
  
    

  		





  
  192
  
    
A network resource URN (NRURN) is defined by the following rules. These rules follow Augmented BFR~\cite{rfc5234} format.


  		





  
  193
  
    
\begin{quote}


  		





  
  194
  
    
  \texttt{NRURN = \qq{}urn:ogf:network:\qq{} DOMAIN \qq{}:\qq{} OPAQUE-PART } \\


  		





  
  195
  
    
  \texttt{DOMAIN = *1DP 1*(ALPHA / DIGIT / \qq{}-\qq{} / \qq{}.\qq{})} ; Domain name \\


  		





  
  196
  
    
  \texttt{DP = \qq{}domain=\qq{}} ; domain prefix (deprecated) \\


  		





  
  197
  
    
  \texttt{OPAQUE-PART = *(ALPHA / DIGIT / OTHER)} \\


  		





  
  198
  
    
  \texttt{OTHER = ALLOWED / RESERVED} \\


  		





  
  199
  
    
  \texttt{ALLOWED = \qq{}+\qq{} / \qq{},\qq{} / \qq{}-\qq{} / \qq{}.\qq{} / \qq{}:\qq{} / \qq{};\qq{} / \qq{}=\qq{} / \qq{}\underscore\qq{}} \\


  		





  
  200
  
    
  \texttt{RESERVED = \qq{}!\qq{} / \qq{}\$\qq{} / \qq{}\q{}\qq{} / \qq{}(\qq{} / \qq{})\qq{} / \qq{}*\qq{} / \qq{}@\qq{}}


  		





  
  201
  
    
\end{quote}


  		





  
  202
  
    

  		





  
  203
  
    
\texttt{ALPHA} and \texttt{DIGIT} are defined by \cite{rfc5234}, \texttt{OTHER} is equal to \texttt{<other>} as defined by~\cite{rfc2141}.


  		





  
  204
  
    

  		





  
  205
  
    
\texttt{OPAQUE-PART} \MUSTNOT{} contain percentage-encoded characters (\texttt{\qq\%\qq{} HEXDIG HEXDIG}). The following pchar characters which are allowed in an URI are never allowed in a network resource URN: \texttt{\qq\%\qq}, \texttt{\qq/\qq}, \texttt{\qq{}?\qq}, \texttt{\qq\#\qq}, \texttt{\qq\&\qq}, and \texttt{\qq\textasciitilde\qq}. 


  		





  
  206
  
    

  		





  
  207
  
    
\texttt{DP} is a prefix (\texttt{\qq{}domain=\qq{}}) of the \texttt{DOMAIN} 


  		





  
  208
  
    
for backward compatibility with a previous syntax (see section~\ref{sec:perfsonar}). 


  		





  
  209
  
    
It is deprecated and \SHOULDNOT{} be used for the assignment of new network resource URNs.


  		





  
  210
  
    

  		





  
  211
  
    
\texttt{DOMAIN} is domain name of the URN assigning organisation in LDR format~\cite{rfc5890}. 


  		





  
  212
  
    
Valid examples are \texttt{example.net} and \texttt{example.xn--jxalpdlp}. 


  		





  
  213
  
    
\texttt{\underscore{}http.\underscore{}tcp.example.net} is an invalid example.


  		





  
  214
  
    

  		





  
  215
  
    
The full lengths of \texttt{NRURN} \MUSTNOT{} exceed 255 bytes.


  		





  
  216
  
    

  		





  
  217
  
    
\texttt{OPAQUE-PART} is opaque, and \MUSTNOT{} be parsed or interpreted by any 


  		





  
  218
  
    
organisation except for the organisation that assigned the URN.


  		





  
  219

\subsection{Encoding}


  		





  
  221
  
    

  		





  
  222
  
    
No percentage-encoded bytes are allowed, as these may prompt users to give meaningful names to network resource URNs. This is discouraged, as described in section~\ref{sec:persistence}.


  		





  
  223
  
    

  		





  
  224
  
    
\subsection{Rules for Lexical Equivalence}


  		





  
  225
  
    

  		





  
  226
  
    
Network resource URNs are lexical equivalent if and only if they are byte-equivalent after case normalisation.


  		





  
  227
  
    

  		





  
  228
  
    
Consider the following URNs:


  		





  
  229
  
    

  		





  
  230
  
    
  1- \texttt{urn:ogf:network:example.net:path-glif-0418} \\


  		





  
  231
  
    
  2- \texttt{UrN:oGf:NeTwOrK:eXaMpLe.NeT:pAtH-gLiF-0418} \\


  		





  
  232
  
    
  3- \texttt{URN:OGF:NETWORK:EXAMPLE.NET:PATH-GLIF-0418}


  		





  
  233
  
    
  4- \texttt{urn:ogf:network:domain=example.net:path-glif-0418} \\


  		





  
  234
  
    

  		





  
  235
  
    
URNs 1, 2, and 3 are lexically equivalent to each other. 


  		





  
  236
  
    
URN 4 is not equivalent to any other listed URN.

\subsection{Assignment}


  		





  
  239
  
    

  		





  
  240
  
    
The characters defined in \texttt{RESERVED} \SHOULDNOT{} be used in assignment 


  		





  
  241
  
    
of network resources URNs, and are reserved for future use. 


  		





  
  242
  
    
Only characters in \texttt{ALPHA / DIGIT / ALLOWED} \SHOULD{} be used in the \texttt{OPAQUE-PART}.


  		





  
  243

\subsection{Validation}


  		





  
  245
  
    

  		





  
  246
  
    
No specific validation service or resolution service is defined in this document.


  		





  
  247
  
    

  		





  
  248
  
    
A recipient of a network resource URN \MUST{} either accept or reject 


  		





  
  249
  
    
the received URN, but \MUSTNOT{} rewrite or normalise it. 


  		





  
  250
  
    
In a data exchange, the URN \MUST{} retain the same capitalisation. 


  		





  
  251
  
    
An network resource URN that does not follow the specified syntax \SHOULD{} be rejected.


  		





  
  252
  
    

  		





  
  253
  
    
For example, percentage-decoding of the URN as described in section 6.2.2.2. 


  		





  
  254
  
    
of~\cite{rfc3986} \MUSTNOT{} take place.

\section{Namespace Considerations}


  		





  
  257
  
    

  		





  
  258
  
    
\subsection{Scope}


  		





  
  259
  
    

  		





  
  260
  
    
The \texttt{urn:ogf:network} namespace is created to allow network 


  		





  
  261
  
    
operators to uniquely define resources in their network and facilitate 


  		





  
  262
  
    
unambiguous exchange of topology data with other network operators.


  		





  
  263
  
    

  		





  
  264
  
    
The only requirement for naming network resources is \emph{persistent} 


  		





  
  265
  
    
administrative ownership of a DNS domain (see section~\ref{sec:persistence}). 


  		





  
  266
  
    
No other central registration is required.


  		





  
  267
  
    

  		





  
  268
  
    
The intended use of the \texttt{urn:ogf:network} namespace is to describe 


  		





  
  269
  
    
logical network resources roughly on OSI layers 1 and OSI layer 2. 


  		





  
  270
  
    
“Logical network resources” intends to mean elements in a functional topology 


  		





  
  271
  
    
description, rather than physical resources. It is expected that a peering 


  		





  
  272
  
    
network is only interested in the functional description of the network, 


  		





  
  273
  
    
not of its (physical) implementation. Nevertheless, this document does 


  		





  
  274
  
    
not forbid the description of other resources, such a physical network 


  		





  
  275
  
    
resources for inventory management.


  		





  
  276
  
    

  		





  
  277
  
    
\subsection{Resource Type Described}


  		





  
  278
  
    

  		





  
  279
  
    
The exact type of resource described by a URN can not and \MUSTNOT{} be 


  		





  
  280
  
    
determined from the syntax of the URN. This information \MUST{} be provided 


  		





  
  281
  
    
by the context or through other means by the data exchange protocol.


  		





  
  282
  
    

  		





  
  283
  
    
Network resources URNs \SHOULD{} identify \emph{manifestations} of a 


  		





  
  284
  
    
network resources --- they should refer to a functional component in a 


  		





  
  285
  
    
network that remains in place for a prolonged period of time. 


  		





  
  286
  
    
New version of the resource \SHOULDNOT{} receive a new identifier.


  		





  
  287
  
    
The change of attributes over time \SHOULD{} be dealt with by a protocol, 


  		





  
  288
  
    
not by a change of the URN.


  		





  
  289
  
    

  		





  
  290
  
    
\subsection{Identifier uniqueness considerations}


  		





  
  291
  
    

  		





  
  292
  
    
URN identifiers \MUST{} be assigned uniquely -- they are assigned to 


  		





  
  293
  
    
at most one resource, and \MUSTNOT{} be re-assigned.


  		





  
  294
  
    

  		





  
  295
  
    
URN assigning organisations \MUST{} follow these requirements before 


  		





  
  296
  
    
assigning URNs to network resources.

\section{Community Considerations}


  		





  
  299
  
    

  		





  
  300
  
    
\subsection{Process of identifier assignment}


  		





  
  301
  
    

  		





  
  302
  
    
Network resource URNs \MAY{} be assigned by any interested party. 


  		





  
  303
  
    
No registration for the URN delegation is required.


  		





  
  304
  
    

  		





  
  305
  
    
The assigning organisation \MUST{} be administrative owner of a domain 


  		





  
  306
  
    
name\~cite{rfc5890}. The assigning organisation \MUST{} be committed to 


  		





  
  307
  
    
remain administrative owner of this domain name for a prolonged period 


  		





  
  308
  
    
of time, at least longer then the expected lifetime of the network 


  		





  
  309
  
    
resources to be identified. It is recommended that a short and stable 


  		





  
  310
  
    
domain name is chosen. The domain name does not need have an actual 


  		





  
  311
  
    
DNS record (e.g. it is allowed to pick “\texttt{network.example.net}” 


  		





  
  312
  
    
if the organisation is administrative owner of “\texttt{example.net}”, 


  		





  
  313
  
    
even if \texttt{network.example.net} does not exist)


  		





  
  314
  
    

  		





  
  315
  
    
The assigning organisation uses the chosen domain name as the 


  		





  
  316
  
    
\texttt{DOMAIN} part of the network resource URN. 


  		





  
  317
  
    

  		





  
  318
  
    
The assigning organisation then assigns \texttt{OPAQUE-PART}s to 


  		





  
  319
  
    
it network resources. The following requirements apply to the 


  		





  
  320
  
    
\texttt{OPAQUE-PART}:


  		





  
  321
  
    
\begin{itemize}


  		





  
  322
  
    
    \item The \texttt{OPAQUE-PART} \MUST{} uniquely define at most one network resource;


  		





  
  323
  
    
    \item The \texttt{OPAQUE-PART} \MUSTNOT{} be re-assigned;


  		





  
  324
  
    
    \item The \texttt{OPAQUE-PART} \SHOULDNOT{} specify any properties of the network resource;


  		





  
  325
  
    
    \item The \texttt{OPAQUE-PART} \MAY{} contain some structure according to some policy internal to the assigning organisation.


  		





  
  326
  
    
\end{itemize}


  		





  
  327
  
    

  		





  
  328
  
    
The reason that the \texttt{OPAQUE-PART} \SHOULDNOT{} contain any properties 


  		





  
  329
  
    
is because a URN \MUST{} be persistent: it \MUSTNOT{} change, even after the 


  		





  
  330
  
    
properties of the described resource change.


  		





  
  331
  
    
Naming these properties in the URN gives a false sense of meaning to the URN. 


  		





  
  332
  
    
Peer may indadvertedly assume the identifier describes certain 


  		





  
  333
  
    
properties, and act upon that, even if the properties have long changed.


  		





  
  334
  
    

  		





  
  335
  
    
Good examples of URNs:


  		





  
  336
  
    
\begin{quote}


  		





  
  337
  
    
\texttt{urn:ogf:network:example.net:9ad7ef-mcasip-139284} \\


  		





  
  338
  
    
\texttt{urn:ogf:network:example.net:link:925-175} \\


  		





  
  339
  
    
\texttt{urn:ogf:network:example.net:port:2011:129}


  		





  
  340
  
    
\end{quote}


  		





  
  341
  
    

  		





  
  342
  
    
Not so good examples of URNs:


  		





  
  343
  
    
\begin{quote}


  		





  
  344
  
    
\texttt{urn:ogf:network:example.net:link:24x7-protected:925-175} \\


  		





  
  345
  
    
\texttt{urn:ogf:network:example.net:path:US\underscore{}CHI-NL\underscore{}AMS-3937}


  		





  
  346
  
    
\end{quote}


  		





  
  347
  
    

  		





  
  348
  
    
A useful syntax for \texttt{OPAQUE-PART} is 


  		





  
  349
  
    
\texttt{<type>:<year of creation>:<sequence number>}, 


  		





  
  350
  
    
e.g. \texttt{port:2011:129}.


  		





  
  351
  
    

  		





  
  352
  
    
While \texttt{port:2011:129} contains attributes (type and year of 


  		





  
  353
  
    
creation), these may be acceptable as they will never change. 


  		





  
  354
  
    
\texttt{link:24x7-protected:925-175} contains attributes about the type of link, 


  		





  
  355
  
    
which may change in the future, and is therefore not a good URN. 


  		





  
  356
  
    
\texttt{link:US\underscore{}CHI-NL\underscore{}AMS-3937} is not a good URN. 


  		





  
  357
  
    
It does contain the end points of a path, which are unlikely to change. 


  		





  
  358
  
    
However, if the path is actually an Ethernet LAN, it is possible to add 


  		





  
  359
  
    
another end-point, changing these properties. The network domains along 


  		





  
  360
  
    
the path may use this identifier for monitoring and do not accept a 


  		





  
  361
  
    
change in identifier. For that reason, it is best never to add attributes 


  		





  
  362
  
    
to a URN identifier.


  		





  
  363
  
    

  		





  
  364
  
    
\subsection{Identifier persistence considerations}


  		





  
  365
  
    
\label{sec:persistence}


  		





  
  366
  
    

  		





  
  367
  
    
\cite{rfc3406} requires that URNs \MUSTNOT{} be re-assigned.


  		





  
  368
  
    

  		





  
  369
  
    
In practice, this is a meaningless requirement. Perhaps in five 


  		





  
  370
  
    
generations from now, URNs will not be used in the first place. 


  		





  
  371
  
    

  		





  
  372
  
    
This document specifies that URNs \MUSTNOT{} get re-assigned 


  		





  
  373
  
    
within the lifetime of the \emph{resource identifier}. 


  		





  
  374
  
    
This is a \textbf{longer period than the lifetime of the resource} itself.


  		





  
  375
  
    
For example, a monitoring system may still use the identifier of a 


  		





  
  376
  
    
resource to display historic monitoring data, long after the resource 


  		





  
  377
  
    
is decommissioned.


  		





  
  378
  
    

  		





  
  379
  
    
Given the lifetime of network resources may last a decade, 


  		





  
  380
  
    
it is reasonable to require that an assigning domain \SHOULD{} validate 


  		





  
  381
  
    
resource identifiers for 5 to 10 year after the decommissioning of 


  		





  
  382
  
    
the resource. It is reasonable to require that an assigning domain 


  		





  
  383
  
    
\MUST{} take measure to ensure that a resource identifier is not 


  		





  
  384
  
    
re-assigned within a couple of decades (20-30 years).


  		





  
  385
  
    

  		





  
  386
  
    
This persistence requirement poses a substantial requirement about 


  		





  
  387
  
    
the stability of the chosen \texttt{DOMAIN} part in the URN. 


  		





  
  388
  
    
\cite{rfc3406} specifically states:


  		





  
  389
  
    
\begin{quote}


  		





  
  390
  
    
“Operationally, there is nothing that prevents a domain name


  		





  
  391
  
    
from being reassigned;  indeed, it is not an uncommon occurrence.


  		





  
  392
  
    
This is one of the reasons that [domain names in a URN] makes a 


  		





  
  393
  
    
poor URN namespace in practice, and is therefore not seriously being


  		





  
  394
  
    
proposed as it stands.”


  		





  
  395
  
    
\end{quote}


  		





  
  396
  
    

  		





  
  397
  
    
In short, the domain must be chosen such that the administrative 


  		





  
  398
  
    
ownership will remain with the current owner for over 30 years after 


  		





  
  399
  
    
assigning the last identifier with that given namespace. This holds 


  		





  
  400
  
    
true even if the network operator changes name, is merged, split or 


  		





  
  401
  
    
files for bankruptcy.


  		





  
  402
  
    

  		





  
  403
  
    
Any organisation that wishes to assign names in the \texttt{urn:ogf:network} 


  		





  
  404
  
    
namespace must to so after due diligence.


  		





  
  405
  
    

  		





  
  406
  
    
The authors of this document have considered using alternative naming strategies for the \texttt{DOMAIN} part, but decided not to do so for these reasons:


  		





  
  407
  
    
\begin{itemize}


  		





  
  408
  
    
    \item There was prior use of domain names in the urn:ogf:network namespace;


  		





  
  409
  
    
    \item The use of autonomous systems (AS) numbers was not possible because not all networks that wanted to assign identifiers has their own AS number;


  		





  
  410
  
    
    \item Creating a separate naming authority would provide too much overhead and hinder uptake of the \texttt{urn:ogf:network} namespace.


  		





  
  411
  
    
\end{itemize}


  		





  
  412
  
    

  		





  
  413
  
    
\section{Examples}


  		





  
  414
  
    

  		





  
  415
  
    
Syntactically valid network resource URNs include:


  		





  
  416
  
    
\begin{quote}


  		





  
  417
  
    
\texttt{urn:ogf:network:example.net:9ad7ef-mcasip-139284} \\


  		





  
  418
  
    
\texttt{urn:ogf:network:example.net:link:925-175} \\


  		





  
  419
  
    
\texttt{urn:ogf:network:example.net:port:2011-07-129} \\


  		





  
  420
  
    
\texttt{urn:ogf:network:domain=example.net:node=packrat:port=eth0}


  		





  
  421
  
    
\texttt{urn:ogf:network:example.net:}


  		





  
  422
  
    
\end{quote}

\section{Security Considerations}


  		





  
  425
  
    

  		





  
  426
  
    
The allowed syntax is so limited that it not expected that similar-looking 


  		





  
  427
  
    
malicious URNs will be an issue. Users should be able to see the 


  		





  
  428
  
    
differences between \texttt{urn:ogf:network:example.c0m:4638I27} 


  		





  
  429
  
    
and \texttt{urn:ogf:network:example.com:4638127}.


  		





  
  430
  
    

  		





  
  431
  
    
Software that takes input from a user \MUST{} ensure that the URN is 


  		





  
  432
  
    
syntactically correct before transmitting it. It \SHOULD{} remove any 


  		





  
  433
  
    
trailing spaces from the user input.


  		





  
  434
  
    

  		





  
  435
  
    
Recipients of a URN \MUSTNOT{} assume that a URN was crafted by the domain


  		





  
  436
  
    
specified in the \texttt{DOMAIN} part of the URN, without a proper validation 


  		





  
  437
  
    
check.


  		





  
  438
  
    

  		





  
  439
  
    
Information in the \texttt{OPAQUE-PART} \MUSTNOT{} be interpreted to have 


  		





  
  440
  
    
any meaning whatsoever. While the originating domain may have included 


  		





  
  441
  
    
meaningful attributes in the URN, these attributes may be out-of-date.


  		





  
  442
  
    

  		





  
  443
  
    
\section{Prior Usage}


  		





  
  444
  
    

  		





  
  445
  
    
URN identifiers in the \texttt{urn:ogf:network} namespace have been in use in 


  		





  
  446
  
    
two communities, with mutually conflicting syntaxes.


  		





  
  447
  
    

  		





  
  448
  
    
\subsection{GLIF Community} %


  		





  
  449
  
    
\label{sec:glif}


  		





  
  450
  
    

  		





  
  451
  
    
The global lambda integrated facility (GLIF) is a community of research and 


  		





  
  452
  
    
education networks. Operators in this community agreed to use unique identifier 


  		





  
  453
  
    
for \emph{lightpaths}, dedicated inter-domain circuits for researchers.


  		





  
  454
  
    

  		





  
  455
  
    
These identifiers take the form:


  		





  
  456
  
    
\begin{quote}


  		





  
  457
  
    
  \texttt{GLOBAL-ID = \qq{}urn:ogf:network:\qq{} DOMAIN \qq{}:\qq{} LOCAL-PART } \\


  		





  
  458
  
    
  \texttt{DOMAIN = 1*(ALPHA / DIGIT / \qq{}-\qq{} / \qq{}.\qq{})} ; Domain name \\


  		





  
  459
  
    
  \texttt{LOCAL-PART = 1*(ALPHA / DIGIT / \qq{}-\qq{} / \qq{}.\qq{})}


  		





  
  460
  
    
\end{quote}


  		





  
  461
  
    

  		





  
  462
  
    
For example:


  		





  
  463
  
    
\begin{quote}


  		





  
  464
  
    
\texttt{urn:ogf:network:canarie.ca:kisti-uninett-glif-001} \\


  		





  
  465
  
    
\texttt{urn:ogf:network:es.net:4005} \\


  		





  
  466
  
    
\texttt{urn:ogf:network:dcn.internet2.edu:6811}


  		





  
  467
  
    
\end{quote}


  		





  
  468
  
    

  		





  
  469
  
    
All identifiers described by the GLIF community are valid network resource URNs.


  		





  
  470
  
    

  		





  
  471
  
    
The syntax is described in \cite{glif-identifiers}.

\subsection{PerfSONAR Community} %


  		





  
  474
  
    
\label{sec:perfsonar}


  		





  
  475
  
    

  		





  
  476
  
    
PerfSONAR is an distributed system for network performance monitoring on 


  		





  
  477
  
    
paths crossing several networks. Much of the perfSONAR protocols are 


  		





  
  478
  
    
standardised by the OGF in the Network Measurement (NM) and Network 


  		





  
  479
  
    
Measurement and Control (NMC) working groups. 


  		





  
  480
  
    
URNs in the \texttt{urn:ogf:network} namespace are used for topology description.


  		





  
  481
  
    

  		





  
  482
  
    
These identifiers take the form:


  		





  
  483
  
    
\begin{quote}


  		





  
  484
  
    
  \texttt{PS-URN = \qq{}urn:ogf:network\qq{} 1DOMAIN-PART *1NODE-PART *1PORT-PART } \\


  		





  
  485
  
    
  \hspace*{2cm}\texttt{*1LINK-PART *1PATH-PART *1SERVICE-PART *1WILDCARD } \\


  		





  
  486
  
    
  \texttt{DOMAIN-PART  = \qq{}:domain=\qq{} 1*DOMAIN} \\


  		





  
  487
  
    
  \texttt{NODE-PART    = \qq{}:node=\qq{} 1*PART-CHAR} \\


  		





  
  488
  
    
  \texttt{PORT-PART    = \qq{}:port=\qq{} 1*PART-CHAR} \\


  		





  
  489
  
    
  \texttt{LINK-PART    = \qq{}:link=\qq{} 1*PART-CHAR} \\


  		





  
  490
  
    
  \texttt{PATH-PART    = \qq{}:path=\qq{} 1*PART-CHAR} \\


  		





  
  491
  
    
  \texttt{SERVICE-PART = \qq{}:service=\qq{} 1*PART-CHAR} \\


  		





  
  492
  
    
  \texttt{DOMAIN = 1*(ALPHA / DIGIT / \qq{}-\qq{} / \qq{}.\qq{})} ; Domain name \\


  		





  
  493
  
    
  \texttt{PART-CHAR = (ALPHA / DIGIT / \qq{}-\qq{} / \qq{}.\qq{} / \qq{}/\qq{} / \qq{}\underscore\qq{})} ; Undefined \\


  		





  
  494
  
    
  \texttt{WILDCARD = \qq{}:*\qq{}} ; Used for queries. Unclear if this is a query or URN.


  		





  
  495
  
    
\end{quote}


  		





  
  496
  
    

  		





  
  497
  
    
For example:


  		





  
  498

  501
  
    
% Bidirectional Link  & \texttt{urn:ogf:network:domain=example.net:link=WASH\underscore{}to\underscore{}ATLA} \\

% Interface           & \texttt{urn:ogf:network:domain=example.net:node=packrat:port=eth0} \\

% Logical Interface   & \texttt{urn:ogf:network:domain=example.net:port=Interface\underscore{}To\underscore{}Geant} \\


  		





  
  504
  
    
% Service             & \texttt{urn:ogf:network:domain=example.net:node=packrat:service=Optical\underscore{}Converter} \\


  		





  
  505
  
    
% Unidirectional Link & \texttt{urn:ogf:network:domain=example.net:node=packrat:port=eth0:link=WASH\underscore{}to\underscore{}ATLA} \\


  		





  
  506
  
    
% Circuit             & \texttt{urn:ogf:network:domain=example.net:path=IN2P3\underscore{}Circuit} \\


  		





  
  507
  
    
% \end{tabular}


  		





  
  508
  
    
\begin{quote}


  		





  
  509
  
    
\texttt{urn:ogf:network:domain=example.net} \\


  		





  
  510
  
    
\texttt{urn:ogf:network:domain=example.net:node=packrat} \\


  		





  
  511
  
    
\texttt{urn:ogf:network:domain=example.net:link=WASH\underscore{}to\underscore{}ATLA} \\

\texttt{urn:ogf:network:domain=example.net:node=packrat:port=eth0} \\

\texttt{urn:ogf:network:domain=example.net:port=Interface\underscore{}To\underscore{}Geant} \\

  515
  
    
\texttt{urn:ogf:network:domain=example.net:node=packrat:port=eth0:link=WASH\underscore{}to\underscore{}ATLA} \\


  		





  
  516
  
    
\texttt{urn:ogf:network:domain=example.net:path=IN2P3\underscore{}Circuit} \\

\texttt{urn:ogf:network:domain=example.net:node=packrat:*}

All identifiers described by the perfSONAR topology service are valid network resource URNs.

 <h1><a name="Benefits"></a>Benefits<a href="#Benefits" class="section_anchor"></a></h1><p>Currently, each element has a global identifier of some kind. For example, layer3 Interfaces have IP addresses and E2EMon links have a global name. However, each new style of global identifier requires some added code in the LS to handle summarization. URNs offer a standard way to uniquely describe these resources without having each type of element or each project defining its own style of identifier. </p><h1><a name="Typical_Use-Case"></a>Typical Use-Case<a href="#Typical_Use-Case" class="section_anchor"></a></h1><p>A user wishes to know what&#x27;s occurring between two hosts. He looks up the traceroute between two hosts. He might then consult the gLS to find the TSs containing information about each of those IPs. Using the TS, he looks up the network resources that those links/interfaces depend on (e.g. the layer2 or layer1 elements under a layer3 link), grabbing the identifier for each resource. He can then look up which MAs contain information on each identifier. </p><h1><a name="URNs"></a>URNs<a href="#URNs" class="section_anchor"></a></h1><p>URNs are globally unique identifiers that can be used to identify network elements no matter what technology they are. </p><p>The identifiers are constructed as Uniform Resource Names (URNs) in the RFC 2141. The namespace for the URN is &quot;ogf&quot; and the subnamespace is &quot;network&quot;. All identifiers must begin with &quot;urn:ogf:network:&quot; in this format. The remainder of the identifier consists of a series of &quot;name&quot; and &quot;value&quot; pairs. These fields provide the hierarchy and the flexibility offered by this schema. </p><p>There are five major fields defined for network entities: </p><ul><li>domain </li><li>node </li><li>port </li><li>link </li><li>path </li><li>service </li></ul><p>The <i>domain</i> field describes the &quot;administrative domain&quot; in which the <i>network</i> element is located. If this field is included, it will always be positioned first. The value is the globally unique DNS name for the domain. While other options exist for this value (e.g. AS number), the simplicity and availability of the former makes this more desirable. </p><p>The <i>node</i> field may be a &quot;host&quot;, &quot;router&quot;, or even a larger  abstraction such as a &quot;site&quot;.  The position of this element will always be second with respect to the <i>domain</i>.  There are no specific rules governing value of this element, allowing maximum flexibility when encoding. </p><p>The <i>port</i> field describes the interface between a <i>node</i> and a <i>link</i> and commonly describes Ethernet interfaces, IP interfaces, names, or listening TCP sockets.  The overall structure of the identifier will be dictated based on whether the interface is physical, logical constructed inside a single device or logically constructed across multiple devices. If the interface is physical or logically inside a single device, the field will appear after the <i>node</i>. If the interface is a logical interface for a domain, the field will appear after the <i>domain</i>. A prudent choice for the value is the physical interface from a networked device in the case of an interface in a physical device or a logical name for logical interfaces in a domain. </p><p>The <i>link</i> field can describe logical or physical <i>link_s, regardless of direction.  Bidirectional </i>link_s (both physical and logical) must appear directly after the <i>domain</i> designation since they will logically connect multiple <i>node_s.  Unidirectional </i>link_s, regardless of direction, will appear after <i>port</i> that is able to &quot;write&quot; that link. As in previous descriptions there is no constraint placed on the value but care should be taken to select a name with meaning to the overall infrastructure. </p><p>The <i>path</i> field can be used for circuits or other named paths. A domain can create an identifier for a <i>path</i> that is completely contained within the domain, or spans multiple domains. In either case, the <i>path</i> field must appear immediately after the <i>domain</i> field of the domain creating the identifier. The value can be anything but must be unique in the domain. </p><p>The <i>service</i> field is used in identifiers for software services offered by network elements. These can be used to describe high-level services like Web Services or low-level services like ICMP responders or optical converters. This field will appear after the field for the element providing the service. For example, a service offered by a domain would appear immediately after the <i>domain</i> field. The value can be anything, but should be a human-readable description of the service provided. </p><h1><a name="Examples"></a>Examples<a href="#Examples" class="section_anchor"></a></h1><p><table class="wikitable"><tr><td style="border: 1px solid #ccc;
