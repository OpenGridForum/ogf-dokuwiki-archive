\section{JobUsageBlock}
2 

  		





  
  3
  
    
This block contains the properties related to job usage. The block is intended to be used for jobs submitted to a batch system.


  		





  
  4
  
    

  		





  
  5

  8
  
    
</ur:JobUsageBlock> 


  		





  
  9
  
    
\end{XMLexample}

\subsection{GlobalJobId}


  		





  
  12
  
    

  		





  
  13
  
    
This property describes the global identity of the job. The property should identify the job globally, such that clashes do not happen accidentally. This could for example be a combination of time stamp, local job id and host name.


  		





  
  14
  
    
\begin{itemize}


  		





  
  15
  
    
\item The GlobalJobId field type MUST be a string.


  		





  
  16
  
    
\end{itemize}


  		





  
  17

\subsection{LocalJobId}


  		





  
  19
  
    

  		





  
  20
  
    
This property describes the local identity of the job. 


  		





  
  21
  
    
\begin{itemize}


  		





  
  22
  
    
\item This element MUST contain the LRMS ID for the job as assigned by the batch system.


  		





  
  23
  
    
\end{itemize}

  26
  
    

  		





  
  27
  
    
This attribute may contain a descriptive of the job. It has to be stressed that user defined job names are often difficult to retrieve from an accounting perspective and are not suitable for reliable accounting purposes.

\subsection{Charge}


  		





  
  30
  
    

  		





  
  31
  
    
This is property may be used for economic accounting purpose. It is the charge applied to the user resource usage.


  		





  
  32
  
    
\begin{itemize}


  		





  
  33
  
    
\item When specifying a charge, the following two attributes MUST be specified:


  		





  
  34
  
    
\begin{itemize}


  		





  
  35
  
    
\item unit – defines the currency used to report the charge.


  		





  
  36
  
    
\item formula – The cost computation formula adopted to compute the applied charge.


  		





  
  37
  
    
\end{itemize}


  		





  
  38
  
    
\end{itemize}

\subsection{Middleware}


  		





  
  57
  
    

  		





  
  58
  
    
The purpose of this element is to mark whether the job was submitted locally or through a grid middleware. At least the values ``local'' and ``grid'' MUST be supported. The description attribute SHOULD be used to give additional information on the used middleware.


  		





  
  59
  
    
\begin{itemize}


  		





  
  60
  
    
\item Middleware MUST be present.


  		





  
  61
  
    
\item The values ``local'' and ``grid'' MUST be supported.


  		





  
  62
  
    
\item The description attribute SHOULD be used.


  		





  
  63
  
    
\end{itemize}

\subsection{TimeInstant}


  		





  
  74
  
    

  		





  
  75
  
    
This element may be used to insert in the records any time instant related to the user payload. Three optional values for the type attribute are defined as they are of common usage by batch systems. The semantic is derived from Torque.


  		





  
  76
  
    
\begin{itemize}


  		





  
  77
  
    
\item TimeInstant SHOULD be present in the record.


  		





  
  78
  
    
\item TimeInstant MAY be present multiple times.


  		





  
  79
  
    
\item The following three types are defined and SHOULD be reported:


  		





  
  80
  
    
\begin{itemize}


  		





  
  81
  
    
\item Ctime - Time job was created


  		





  
  82
  
    
\item Qtime - Time job was queued


  		





  
  83
  
    
\item Etime - Time job became eligible to run


  		





  
  84
  
    
\end{itemize}


  		





  
  85
  
    
\end{itemize}

  88
  
    

  		





  
  89
  
    
This element is used to insert computing benchmarks and normalization factors.


  		





  
  90
  
    
At least one normalization factor MUST be present in the record.

  93

  94
  
    
\item At least the following types should be supported:


  		





  
  95
  
    
\begin{itemize}


  		





  
  96
  
    
\item Si2k – SpecInt2000


  		





  
  97
  
    
\item Sf2k – SpecFloat2000


  		





  
  98
  
    
\item HEPSPEC – HEPSpec


  		





  
  99
  
    
\end{itemize}

\subsection{Status}


  		





  
  103
  
    

  		





  
  104
  
    
This property will represent the completion status of the job. For example, this may


  		





  
  105
  
    
represent the exit status of an interactive running process or the exit status from the batch


  		





  
  106
  
    
queuing system’s accounting record. The semantic meaning of status is site dependent.


  		





  
  107
  
    
\begin{itemize}


  		





  
  108
  
    
\item The value of Status MUST be of type String.


  		





  
  109
  
    
\item The Status property MUST exist in the record.


  		





  
  110
  
    
\item The Status property MUST support the following values:


  		





  
  111
  
    
\begin{itemize}


  		





  
  112
  
    
\item aborted – A policy or human intervention caused the job to cease execution.


  		





  
  113
  
    
\item completed – The execution completed.


  		





  
  114
  
    
\item failed – Execution halted without external intervention.


  		





  
  115
  
    
\item held – Execution is held at the time this usage record was generated.


  		





  
  116
  
    
\item queued – Execution was queued at the time this usage record was generated.


  		





  
  117
  
    
\item started – Execution started at the time this usage record was generated.


  		





  
  118
  
    
\item suspended – Execution was suspended at the time this usage record was generated.


  		





  
  119
  
    
\end{itemize}


  		





  
  120
  
    
\item The Status property MAY support other values, as agreed upon within the implementation context.


  		





  
  121
  
    
\end{itemize}
